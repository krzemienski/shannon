## Tasks

### Task 1 — Apply the WINNING filter

The WINNING filter classifies each skill against 5 questions. A skill earns 1 point per "yes":

- **W** — does it serve a v0.1.0 pillar directly? (uses pillars column from matrix)
- **I** — is it Iron Rule-aligned (validation, evidence, no-mocking)? (any skill whose own discipline matches the Iron Rule)
- **N** — is it Newer than v5 in a meaningful sense? (was substantially rewritten in v7 or imported from anthropic-skills)
- **N** — Necessary for the SDK or Tmux harness to function? (infrastructure for Pillar 5)
- **G** — is it Generally useful across multiple workflows? (vs. niche)

Scoring guideline (NOT a hard rule):
- 4-5 points → strong KEEP candidate
- 2-3 points → review case-by-case; likely KEEP if `role: core` from 01-01, likely CUT if `role: tangent`
- 0-1 points → strong CUT candidate

For each skill, record the WINNING score + verdict.

**Action:** Read `evidence/categorization-matrix.md`. For each of the 67 rows, compute the WINNING score and provisional verdict. Write `evidence/winning-scores.md`:

```markdown
| skill | role (from 01-01) | W | I | N1 | N2 | G | score | provisional verdict |

| meta-judge-rubric | core | ✓ | ✓ | ✓ | ✗ | ✓ | 4 | KEEP |
| algorithmic-art | tangent | ✗ | ✗ | ✗ | ✗ | ✗ | 0 | CUT |
| ... |
```

**Verify (gate):**

**Action:** Review the KEEP set from Task 1. For each pair/triplet that overlaps:

The merged skill's content needs Phase 1.3 attention (during execution) — its body will need updates to absorb the merged-in capabilities. For now, just decide what merges with what.
