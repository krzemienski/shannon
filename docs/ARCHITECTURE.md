# Architecture

## 5 pillars

Shannon's design is organized around 5 invariants the framework enforces:

| # | Pillar | What it means | Where it lives |
|---|---|---|---|
| 1 | Embedded sub-agent skills | Sub-agents carry their skills inline; spawning is reliable | `agents/<name>/AGENT.md` (inlined by `build/embed-skills.py`) |
| 2 | Orchestration | Single-message multi-Task dispatch with judge confirmation | `Skill: dispatch-parallel`, `Skill: multi-agent-patterns`, `/shannon:dispatch` |
| 3 | Iron Rule validation | Real-system evidence on disk; no mocks/stubs/test files | `Skill: functional-validation`, `Skill: no-fakes-discipline`, `block-fab-files` hook |
| 4 | Meta-judge consensus | Rubric YAML before any judge; debate on disagreement | `Task: meta-judge`, `Skill: judge`, `Skill: consensus-engine` |
| 5 | Self-instrumented | The plugin observes itself; `doctor.py` is mechanical | `/shannon:doctor`, `scripts/doctor.py`, `Skill: observability-report` |

## Architecture C — the central design call

**The problem with v7**: subagents were spawned with no guarantee that their declared skills would activate. Skill triggers are pattern-matched against user prompts; spawned subagents have their own prompt context with different patterns. Claims like "the agent will use the functional-validation skill" had no on-disk enforcement.

**The Architecture C fix**: bundle each agent's skills directly into the agent's spawn context.

The build step (`build/embed-skills.py`) does two things per agent:

1. **Copies** `skills/<X>/` → `agents/<agent>/_built/skills/<X>/` (for runtime Read-tool access to references/)
2. **Inlines** the SKILL.md body into `agents/<agent>/AGENT.md` between sentinels:

```markdown
<!-- BEGIN EMBEDDED SKILLS — generated by build/embed-skills.py -->

## Embedded skill: `judge`

[full body of skills/judge/SKILL.md goes here]

---

## Embedded skill: `consensus-engine`

[full body of skills/consensus-engine/SKILL.md goes here]

<!-- END EMBEDDED SKILLS -->
```

When Claude Code spawns the agent via `Task(subagent_type="meta-judge")`, the loader reads AGENT.md as the agent's system prompt — including the inlined skills.

**Proof on disk**: in Plan 02-03 smoke test, we moved `skills/judge/` away to `skills/judge.disabled.smoke/`; the md5 of `agents/meta-judge/AGENT.md` was unchanged. The embedded content is independent of canonical-skill presence. See `.planning/phases/02-agent-embedding/evidence/embedded-loading-proof.md`.

## The 7 hooks

| Hook | Event | Consumed by skills |
|---|---|---|
| `block-fab-files` | PreToolUse:Write\|Edit\|MultiEdit | no-fakes-discipline |
| `pre-edit-discipline` | PreToolUse:Edit\|MultiEdit\|Write | (advisory) |
| `subagent-governance` | PreToolUse:Task\|Agent | dispatch-parallel, multi-agent-patterns, team-coordinator |
| `evidence-gate` | PreToolUse:TaskUpdate | evidence-gate, completion-gate, refusal-discipline |
| `post-action-discipline` | PostToolUse:Bash\|Edit\|Write\|MultiEdit | no-fakes-discipline, evidence-gate, completion-gate, functional-validation |
| `observability` | SessionStart + UserPromptSubmit + PostToolUse:* | (passive) |
| `stop-semantics` | Stop | team-coordinator |

The `required_hooks` field in skill frontmatter declares the contract. `scripts/doctor.py` cross-checks: 12 declared dependencies across 8 skills, 0 mismatches.

## The 20 commands

The user-facing surface. Each command routes through agents (via `Task: <agent>`) and skills (via `Skill: <name>`). All 33 skills and all 9 agents are referenced at least once across the 20 commands — 0 phantom references (verified by `scripts/doctor.py` skill-body-refs + agent-body-refs checks).

See [docs/SKILLS_CATALOG.md](SKILLS_CATALOG.md) and `commands/*.md` for the per-command details.

## How a workflow flows through the layers

Example: user runs `/shannon:cook "Add a /health endpoint"`.

1. **Layer 3 (Command)** — `commands/cook.md` body parses arguments; identifies this as a brief (not a plan path); routes to `Task: planner`.
2. **Layer 2 (Agent — planner)** — planner has `plan-author`, `interview-framework`, `goal-condition-architect`, `spec-workflow`, `create-meta-prompts`, `gepetto` embedded. Conducts intake; produces `plans/<date>-add-health/plan.md` + phase files.
3. **Layer 3 (Command, continued)** — cook routes the plan into `Task: executor`.
4. **Layer 2 (Agent — executor)** — executor has `dispatch-parallel`, `functional-validation`, `reflect`, `codebase-analysis` embedded. Executes each phase; spawns parallel sub-tasks where appropriate.
5. **Layer 4 (Hook — block-fab-files)** — if executor attempts a `tests/foo.test.js` Write, the hook fires with exit 2 and stderr `[shannon] block-fab-files: refusing Write...`. The Write is refused.
6. **Layer 4 (Hook — post-action-discipline)** — after every Bash build command, this hook reminds that compilation ≠ functional validation.
7. **Layer 3 (Command, continued)** — cook routes the executor output to `Skill: functional-validation` for evidence capture.
8. **Layer 4 (Hook — evidence-gate)** — when executor calls `TaskUpdate(status=completed)`, this hook checks for fresh evidence in `e2e-evidence/`. If absent, exit 2 with reminder.
9. **Layer 3 (Command, continued)** — cook routes to `Skill: completion-gate` for the mechanical final check.
10. **Layer 1 (Skill — refusal-discipline)** — if any gate cited a blocker, `refusal-discipline` writes `REFUSAL.md` with specific cited blockers. No COMPLETE verdict.

The user sees either:
- A `COMPLETE` verdict with cited evidence files, OR
- A `REFUSAL.md` with specific blockers and remediation steps.

Never a "looks good to me" handwave.
